Chapter 2: How is a logger's level decided? [10]

[10]

Based on my blog post: 自作のPythonライブラリでloggingを使うとき、ロガーにsetLevelしない？ ログレベルNOTSETのロガーの動きを理解する

❓ What is the level of the httpxyz logger? [8]
logger = logging.getLogger("httpxyz")

DEBUG, INFO, WARNING, ERROR or CRITICAL? 🤔

NOTSET

When a logger is created, the level is set to NOTSET

Logger.setLevel()

>>> import logging
>>> httpxyz_logger = logging.getLogger("httpxyz")
>>> logging.getLevelName(httpxyz_logger.level)
'NOTSET'

What does NOTSET mean?

if a logger has a level of NOTSET, its chain of ancestor loggers is traversed until either an ancestor with a level other than NOTSET is found, or the root is reached. (Logger.setLevel())

Ancestor loggers?

📌Logger hierarchy

Loggers have parent-child relationships determined by their names

Names can be dot-separated

Logger names can be anything you want, and indicate the area of an application in which a logged message originates. (Advanced Logging Tutorial)

logging.getLogger("spam")
logging.getLogger("spam.ham")

Parent-child relationships

logger = logging.getLogger("spam.ham")

  • The parent of the spam.ham logger is the spam logger

  • Every logger ultimately descends from the root logger (logging.getLogger() with no arguments)

Parent-child relationships

Use __name__ as the logger name 🏃‍♂️

logger = logging.getLogger(__name__)

  • This is a "good convention to use when naming loggers" (Advanced Logging Tutorial) [11]

[11]

Because you can configure logging by touching the logger for an upper-level module. Today's discussion applies for the same logic.

💡Change the root logger's level

When you change the level of the top parent logger, children loggers' effective levels can also change

Effective level: getEffectiveLevel()

Otherwise, the hierarchy is traversed towards the root until a value other than NOTSET is found, and that value is returned.

The root logger's level is WARNING

>>> import logging
>>> root_logger = logging.getLogger()
>>> logging.getLevelName(root_logger.level)
'WARNING'

The same WARNING as the root logger

>>> httpxyz_logger = logging.getLogger("httpxyz")
>>> logging.getLevelName(httpxyz_logger.getEffectiveLevel())
'WARNING'

When the root logger's level changes

>>> root_logger.setLevel(logging.DEBUG)
>>> logging.getLevelName(root_logger.level)
'DEBUG'

The child logger's effective level has changed

>>> logging.getLevelName(httpxyz_logger.getEffectiveLevel())
'DEBUG'

Summary🥟 how is a logger's level decided?

  • A library logger's level is NOTSET

    • If you do nothing, its effective level is the same WARNING as the root logger

  • Changing the root logger's level also changes descendants' effective levels when their own levels are ``NOTSET`` (we will use this💡)

Chapter 3: Log through the root logger - propagate

In logger ancestor chain, there is one more important thing besides the NOTSET level

The logger's propagate attribute

If this attribute evaluates to true, events logged to this logger will be passed to the handlers of higher level (ancestor) loggers, in addition to any handlers attached to this logger.

After getLogger(), propagate is True by default

>>> import logging
>>> httpxyz_logger = logging.getLogger("httpxyz")
>>> httpxyz_logger.propagate
True

Logger parent-child relationships (again)

propagate

  • A method at or above the httpxyz logger's logging level is called

  • That LogRecord is also passed to the parent logger's handlers (propagation)

With propagate, a LogRecord goes to the parent logger's handlers

Summary🥟 logging through the root logger

  • A library logger's effective level, for example HTTPXYZ, is the same as the root logger's level when its own level is NOTSET

  • A logger created by getLogger() has propagate=True by default (it propagates to the root logger)

  • You can log by setting the root logger's level and attaching handlers (with filters) and formatters to it!

Implementation of basicConfig()
def basicConfig(**kwargs):

    with _lock:
        if len(root.handlers) == 0:

If a library configures handlers on the root logger, calling basicConfig from library-user code later does nothing 😭

Functions such as logging.warning() also call basicConfig()

if the root logger has no handlers, then basicConfig() is called, prior to calling debug on the root logger.

logging.debug()

When no handler is added to a logger

The event is output using a ‘handler of last resort’, stored in lastResort.

The handler’s level is set to WARNING, (omitted)

What happens if no configuration is provided

Use NullHandler if you do not want the last resort handler

If for some reason you don’t want these messages printed in the absence of any logging configuration, you can attach a do-nothing handler to the top-level logger for your library.

Configuring Logging for a Library

Do not attach anything except NullHandler 🙅‍♂️

class InterceptHandler(logging.Handler):
    def emit(self, record: logging.LogRecord) -> None:
        try:
            level: str | int = logger.level(record.levelname).name
        except ValueError:
            level = record.levelno

        frame, depth = inspect.currentframe(), 0
        while frame:
            filename = frame.f_code.co_filename
            is_logging = filename == logging.__file__
            is_frozen = "importlib" in filename and "_bootstrap" in filename
            if depth > 0 and not (is_logging or is_frozen):
                break
            frame = frame.f_back
            depth += 1

        logger.opt(depth=depth, exception=record.exc_info).log(
            level, record.getMessage()

  • A logger's NOTSET level: use it with the same effective level as an ancestor logger
