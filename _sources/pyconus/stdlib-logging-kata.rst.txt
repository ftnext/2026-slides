Logging levels [#level-table-ref-en]_ 🏃‍♂️

    * - Level
      - Method
      - When it’s used
    * - DEBUG
      - ``debug()``
      - Detailed information, typically of interest only when diagnosing problems.
    * - INFO
      - ``info()``
      - Confirmation that things are working as expected.

.. revealjs-break::
    :notitle:

    * - WARNING
      - ``warning()``
      - An indication that something unexpected happened, (omitted)
    * - ERROR
      - ``error()``
      - Due to a more serious problem, the software has not been able to perform some function.
    * - CRITICAL
      - ``critical()``

.. _Logger.setLevel(): https://docs.python.org/3/library/logging.html#logging.Logger.setLevel

**NOTSET**
--------------------------------------------------

    When a logger is created, the level is set to ``NOTSET``

`Logger.setLevel()`_

.. code-block:: pycon

    >>> import logging
    >>> httpxyz_logger = logging.getLogger("httpxyz")
    >>> logging.getLevelName(httpxyz_logger.level)
    'NOTSET'

What does ``NOTSET`` mean?
--------------------------------------------------

   if a logger has a level of ``NOTSET``, its chain of ancestor loggers is traversed until either an ancestor with a level other than ``NOTSET`` is found, or the root is reached. (`Logger.setLevel()`_)

Ancestor loggers?

📌Logger hierarchy
======================================================================

Loggers have **parent-child relationships determined by their names**

Names can be **dot-separated**
--------------------------------------------------

    Logger names can be anything you want, and indicate the area of an application in which a logged message originates. (`Advanced Logging Tutorial <https://docs.python.org/3/howto/logging.html#advanced-logging-tutorial>`__)

.. code-block:: python

    logging.getLogger("spam")
    logging.getLogger("spam.ham")

Parent-child relationships
--------------------------------------------------

.. code-block:: python

    logger = logging.getLogger("spam.ham")

* The **parent** of the ``spam.ham`` logger is the ``spam`` logger
* Every logger ultimately descends from the *root logger* (``logging.getLogger()`` with no arguments)

Summary🥟 how is a logger's level decided?
======================================================================

* A library logger's level is ``NOTSET``

    * If you do nothing, its effective level is the same ``WARNING`` as the root logger

* **Changing the root logger's level also changes descendants' effective levels when their own levels are ``NOTSET``** (we will use this💡)

Chapter 3: Log through the root logger - ``propagate``
======================================================================

In logger ancestor chain, there is one more important thing besides the ``NOTSET`` level

The logger's `propagate`_ attribute
========================================================================================================================

    If this attribute evaluates to true, events logged to this logger will be passed to the handlers of higher level (ancestor) loggers, in addition to any handlers attached to this logger. 

After ``getLogger()``, ``propagate`` is ``True`` by default
------------------------------------------------------------

.. code-block:: pycon

    >>> import logging
    >>> httpxyz_logger = logging.getLogger("httpxyz")
    >>> httpxyz_logger.propagate
    True

Summary🥟 logging through the root logger
======================================================================

* A library logger's effective level, for example HTTPXYZ, is the same as the root logger's level when its own level is ``NOTSET``
* A logger created by ``getLogger()`` has ``propagate=True`` by default (it propagates to the root logger)
* You can log by **setting the root logger's level and attaching handlers (with filters) and formatters** to it!

Chapter 4: Logging implementations with room to improve, and how to deal with them
==================================================================================

Case 2: Do not touch the root logger **in a library**
======================================================================

DON'T🙅‍♂️ call ``basicConfig()`` in a library

The secret of ``basicConfig()``
----------------------------------------------------------------------

    This function does nothing if the root logger already has handlers configured,

`logging.basicConfig()`_

Experiment script
--------------------------------------------------

.. code-block:: python

    import logging

    # Simulates library code imported first
    logging.basicConfig(level=logging.INFO)
    logging.info("as expected")
    logging.warning("a little dangerous")

    # Simulates configuration by the library user
    logging.basicConfig(level=logging.WARNING)  # does not work😭
    logging.info("as expected")
    logging.warning("a little dangerous")

Configure the root logger "only once"
--------------------------------------------------

.. code-block:: python
    :caption: `Implementation of basicConfig() <https://github.com/python/cpython/blob/v3.14.5/Lib/logging/__init__.py#L2083>`__

    def basicConfig(**kwargs):

        with _lock:
            if len(root.handlers) == 0:
                ...
            # does nothing if handlers already exist

If a library configures handlers on the root logger, **calling basicConfig from library-user code later does nothing** 😭

Functions such as ``logging.warning()`` also call ``basicConfig()``
----------------------------------------------------------------------

    if the root logger has no handlers, then ``basicConfig()`` is called, prior to calling ``debug`` on the root logger.

`logging.debug() <https://docs.python.org/3/library/logging.html#logging.debug>`__

Do not use them in libraries, just like ``basicConfig()``
------------------------------------------------------------

.. code-block:: python
    :caption: `Implementation of logging.debug() <https://github.com/python/cpython/blob/v3.14.5/Lib/logging/__init__.py#L2199>`__

    def debug(msg, *args, **kwargs):
        if len(root.handlers) == 0:
            basicConfig()
        root.debug(msg, *args, **kwargs)

Workaround: the ``force`` argument of ``basicConfig()``

When no handler is added to a logger
--------------------------------------------------

    The event is output using a ‘handler of last resort’, stored in ``lastResort``.

    The handler’s level is set to ``WARNING``, (omitted)

`What happens if no configuration is provided <https://docs.python.org/3/howto/logging.html#what-happens-if-no-configuration-is-provided>`__

Use ``NullHandler`` if you do not want the last resort handler
--------------------------------------------------------------

    If for some reason you *don’t* want these messages printed in the absence of any logging configuration, you can attach a do-nothing handler to the top-level logger for your library.

`Configuring Logging for a Library <https://docs.python.org/3/howto/logging.html#configuring-logging-for-a-library>`__

Do not attach anything except ``NullHandler`` 🙅‍♂️
------------------------------------------------------------

* Output from the library logger's handler
* Propagation to the parent root logger, where the root logger's handler also processes it: **duplicate** output

🌯Back to the first question: can you use Python's stdlib ``logging`` with confidence?
================================================================================================

* A logger's ``NOTSET`` level: use it with **the same effective level as an ancestor logger**
* Propagation to the root logger: **output through the root logger's handlers**

Summary🌯: Understanding Python's logging: Combine components like Lego blocks!
------------------------------------------------------------------------------------------------

* Introduced 1 **pattern** derived from ``NOTSET`` and ``propagate``: logging through the root logger
* The **library user** logs in the **format and destination** they want

🌯To **library authors**: "should only call loggers"
------------------------------------------------------------

* You only want to **record events**
* You do not care about log format or destination, assuming users configure those freely
* 👉"Do not touch the root logger in a library"

Even in **local scripts**, not only libraries like HTTPXYZ
------------------------------------------------------------

.. code-block:: python
    :caption: Configure the root logger in one place; inside functions, only call loggers

    def awesome():
        # logger call only (recording an event)
    def fabulous():
        # logger call only (recording an event)
    def main():
        logging.basicConfig(...)  # level, handlers, and formatters on the root logger
        awesome()
        fabulous()
